 A. Final Judgment

  当前结构总评

  三文件架构在设计意图上是正确的——入口层 / always-on 标准 / 按需扩展 的分层清晰，职责边界基本到位。但 always-on
  层（core-standard.md）承载了过多程序化执行细节，使其实际上充当了"第二入口"而非"核心约束层"。

  最关键问题

  core-standard.md §6（Verification & Definition of Done）过于厚重。Required Gates、Applicability、Review Policy、Review Requirements、Completion Rule、Completion Contract
  这六块内容加起来约 800+ 字，本质是一个"状态报告状态机"。这种东西写在 always-on 层，会导致：
  - 每个任务都要套这个壳，降低遵循意愿
  - 流程细节无法按需启用/跳过
  - review 结果没有强制性验证手段，写了等于没写

  推荐方向

  中度重构（非小修收缩，也非推翻重来）：
  - core-standard.md 压缩到 600 行以内，只保留真正的约束规则
  - §6 拆为 2 个 skill（verification-before-completion + receiving-code-review）
  - §4 Task Framing 中的 L0/L1/L2 定义保留，具体 state 流程迁移到 hooks
  - §8 Git Rules 中 force push 等高风险操作改为 hook 强制约束
  - §9 Completion Contract 改为1行原则 + skill 引用

  2. 从 core-standard.md §6 Review 部分提取，生成或引用 receiving-code-review skill（检查 superpowers 下是否已存在）
  3. 从 orchestration-extension §8 提取，生成 systematic-debugging skill
  4. 将 finish-branch.md command 升格为 skill

  风险：中等——需要确保 skill 边界清晰，不产生循环依赖

  验收标准：
  - 每个 skill 有独立文件，符合 skills/<name>/SKILL.md 结构
  - Phase 1 删减的内容已被 skill 承接
  - 原引用位置改为 "See skill: xxx"

  Phase 3: 引入 hooks 做确定性控制

  目标：将高风险操作的文字规则转为确定性 hook 拦截

  改动范围：
  - 在 ~/.claude/settings.json 中配置 hooks
  - 新建 force-push-guard、branch-delete-guard 等 hook
  - 移除 core-standard.md §8 中"never force push"等文字规则（因为 hook 已强制执行）

  风险：中等——hook 配置错误可能误拦正常操作；需要充分测试

  验收标准：
  - git push --force 到 main 被 hook 拦截并要求确认
  - 删除未合并分支被拦截
  - Hook 配置有对应的 fallback 机制（允许 bypass）

  Phase 4: 建立 memory 承接经验性知识

  目标：将积累性、经验性、场景差异性知识迁移到 memory

  改动范围：
  - 新建 memory/completion-claims.md — 各种场景下的完成报告样本
  - 新建 memory/review-evidence.md — review 通过/失败/阻断的实际案例
  - 新建 memory/delegation-patterns.md — 委托决策的实际经验

  风险：低——memory 是纯知识积累，不影响执行

  验收标准：
  - memory 文件有实际内容，不是空模板
  - memory 内容在新的相应场景中被实际使用
  - Phase 1-3 的删减内容不产生知识丢失

  F. Highest-Risk Anti-Patterns

  1. "格式完整性幻觉"（Over-Institutionalized Completeness）

  最危险：core-standard.md §6 + §9 的 Completion Contract 体系

  这不是真正的约束，而是一个"看起来很严谨"的格式模板。问题是：
  - 每次都要求完整填写 7 个字段，即使 L0 任务也如此
  - review evidence 无法被强制验证，写了也可能不符合要求
  - 格式要求降低了实际遵循度

  避免方法：把格式要求降到最低，把强制性留给 skill 和 hook

  2. "Hook 缺失导致的假约束"

  最危险：core-standard.md §8 Git Rules + §7 Security

  "No force push to main"写了等于没写，因为没有 hook 强制执行。用户在高压下会忘记，Claude 在连续操作中会自动跳过。

  避免方法：确定性约束必须用 hook 表达，文字规则只用于描述 hook 的意图

  3. "两层 always-on 标准"

  orchestration-extension.md 的文件名和体量让它实际上成为了第二个"always-on 标准"。§1 明确定义了 on-demand，但文件内部的厚度（9个章节，39个搜索到的
  sections）会让人每次都去翻它，而不是只在需要时才查。

  避免方法：将扩展指南真正按需化——减少章节、增加入口明确性、不在 on-demand 文件里放执行协议

  4. "Task State 的双源风险"

  core-standard.md §4 State Backend 规定"Never keep two authoritative trackers"，但没有 hook 强制这一点。如果 Claude Code 本身维护了任务状态，同时用户又手动维护了另一个
  tracker，两个 tracker 会逐渐漂移。

  避免方法：hook 检测 state conflict，不依赖人的自律

  5. "Completion Claim 的自我参照"

  §6 Verification 定义了 Gate，§9 Completion Contract 要求 Gate 状态被记录，但"Gate 是否真的被满足"的验证本身依赖 Review——而 Review 又依赖 Completion Claim
  的格式正确性。这是一个循环依赖，不是真正的验证链。

  避免方法：打破循环——让 hook 强制 gate 通过，而不是让格式正确的 claim 自证合规
