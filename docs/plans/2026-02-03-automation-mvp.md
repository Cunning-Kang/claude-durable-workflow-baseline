# 自动化 MVP 包（最短可行路径）

**目标**：在“人工完成任务创建、需求澄清、计划审阅”的前提下，使**计划执行与结果判定**进入可自动闭环的最小形态。

---

## 1. 范围与非目标

### 1.1 范围（必须完成）
- 计划步骤可执行化（机器可读）
- 执行器按步骤运行并判定成功/失败
- 质量门自动运行与记录
- 任务状态自动更新（step 级别）
- 统一执行报告（结果可追溯）

### 1.2 非目标（MVP 不做）
- 自动提交与回滚
- 复杂异常自动修复
- 计划自动调整
- 多任务并行执行

---

## 2. MVP 交付物清单

### 交付物 1：计划步骤规范（最低可执行格式）
**目的**：让计划步骤变成可执行单元。

**格式要求（示例）**：
```
### Step 3: Run unit tests
Run: pytest tests/foo/test_bar.py::test_baz -v
Expect: PASS
```

**验收标准**：
- 每个步骤包含 Run 与 Expect 字段
- 计划中不存在无法映射为命令的步骤

---

### 交付物 2：计划校验器（Plan Validator）
**目的**：执行前拦截不可执行计划。

**验收标准**：
- 能识别缺失 Run/Expect 的步骤
- 能输出明确的错误信息与定位
- 验证不通过则不允许执行

---

### 交付物 3：步骤执行器（Step Runner）
**目的**：按步骤顺序自动执行并判定结果。

**行为要求**：
- 逐步执行 Run 命令
- 比对 Expect（PASS/FAIL 或关键字）
- 失败即停并输出报告

**验收标准**：
- 可完整执行计划中所有步骤
- 任一失败步骤能被准确标记

---

### 交付物 4：质量门自动运行
**目的**：确保完成判定基于质量门信号。

**验收标准**：
- 质量门命令来自计划或任务文件
- 成功/失败结果可被机器读取

---

### 交付物 5：任务状态自动更新
**目的**：把执行进度同步回任务系统。

**验收标准**：
- 执行开始 → In Progress
- 每完成一步 → Step N
- 全部通过 → Done
- 失败 → Blocked/Needs Review

---

### 交付物 6：执行报告（统一输出）
**目的**：形成可追溯的执行记录。

**验收标准**：
- 输出 JSON 或 Markdown 报告
- 包含：步骤列表、结果、失败原因、质量门结果

---

## 3. 执行顺序（最短路径）

1. 定义计划步骤规范
2. 实现计划校验器
3. 实现 Step Runner
4. 接入质量门执行
5. 接入任务状态更新
6. 输出执行报告

---

## 4. 入口语句（MVP 触发）

```
按计划执行 TASK-xxx
```

**行为**：触发校验 → 执行 → 质量门 → 状态更新 → 报告输出。

---

## 5. 成功判定（MVP DoD）

1. 任意计划可被校验并执行
2. 质量门成功是完成的唯一条件
3. 失败会停止执行并输出报告
4. 任务状态与执行结果一致

---

## 6. 风险与缓解

**风险**：计划格式不一致 → 执行失败
**缓解**：强制校验器 + 入口语句固定

**风险**：Expect 判定过于粗糙
**缓解**：先支持 PASS/FAIL 与关键字匹配，后续扩展

---

## 7. 输出与归档

- 计划：`docs/plans/`
- 执行报告：建议写入 `docs/reports/` 或 stdout
- 任务状态：`docs/tasks/`

---

## 8. 结论

该 MVP 包提供最短闭环：**计划可执行 → 自动执行 → 质量门判定 → 状态同步**。完成后即可进入“阶段 1：半自动执行”，并为后续高自动化奠定基础。
