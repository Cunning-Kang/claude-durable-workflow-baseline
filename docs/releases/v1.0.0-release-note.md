# v1.0.0 — Durable Workflow Baseline Release

## A. Who This Release Is For

**Primary audience:** Project leads or platform owners setting up a durable workflow baseline for their team.

**Not the audience for:** End users who simply want to use an already-adopted baseline — they consume the hooks and memory output, but do not need to understand the baseline surface itself.

**Relationship to Superpowers:** Superpowers is the primary global control layer. This baseline repo supplies the host-wide constraints and opt-in surfaces that Superpowers then operates within. Do not add generic global skills here that would duplicate Superpowers' planning/execution/review responsibilities.

---

## B. What This Release Is

This is a **baseline source repository** release — a durable starting point for teams adopting Claude Code in the context of durable workflow conventions. It is the authoritative distribution source for host-wide runtime constraints, protocol definitions, opt-in hook templates, and a pre-structured memory skeleton.

This repository is **not**:
- A second control plane or orchestration runtime
- A live default wiring that activates without adoption steps

It ships a stable surface that others can derive from, adopt from, or integrate into project-specific setups.

---

## C. What Is Included

| Surface | Role |
|---------|------|
| `global/` | Host-wide runtime constraints (core-standard, guides, command routing) |
| `baseline/docs/workflow/` | Protocol definitions: execution contract, memory protocol, native task translation, review protocol |
| `distribution/hooks/` | Opt-in hook templates (H-01/H-02/H-03 clusters) organized for adoption |
| `distribution/settings-snippets/` | Opt-in settings snippets for settings.json merge |
| `baseline/memory/` | Memory skeleton: patterns, gotchas, MEMORY.md |
| `docs/reference/` | Reference documentation for maintained surfaces |
| Distribution commands | Available commands include `/init-claude-workflow`, `/new-feature`; see `distribution/commands/` for current inventory |

**Path conventions:**
- `global/` — host-wide surfaces that apply to every project on this host
- `baseline/` — baseline source repo surfaces that need explicit adoption
- `distribution/` — opt-in artifacts ready to copy or merge into a project

## D. What Changed in This Refactor

The v1.0.0 refactor completed four phases:

1. **Structure contraction** — global core was compressed; boundary docs established; audit paths migrated
2. **Bearer layer cleanup** — competing workflow skills removed; protocol docs realigned
3. **Hook implementation** — H-01/H-02/H-03 hook clusters delivered as opt-in templates
4. **Memory boundary** — memory skeleton established with protocol-aligned authority; stale content cleared

The overarching theme: core was narrowed, rule-to-hook migration was completed, memory boundary was formalized, and a release-ready adoption surface was established.

## E. How Adopters Should Start

> **Note:** Known documentation enhancements (BR-01 through BR-07) are post-release additions. They do not block adoption but will improve first-pass clarity over time. See the post-release backlog for context.

**First pass (15 minutes):**
1. Read `global/README.md` — establishes what this repo is and is not
2. Read `baseline/docs/workflow/` — protocol definitions for execution, memory, and review

**Second pass (30 minutes):**
3. Review `distribution/hooks/` and `distribution/settings-snippets/` — understand what opt-in means
4. Review `baseline/memory/MEMORY.md` and `baseline/memory/patterns.md`

**Deep dive (as needed):**
- `docs/reference/` for detailed surface documentation

**Adoption is opt-in.** Hooks require copying to your project directory. Settings snippets require manual merge. Memory content requires filling in per the memory protocol.
